The **Rate Limiter** block protects your application from abuse by limiting how many requests a client can make within a configurable time window.

It is built around a framework-independent core engine with lightweight framework adapters and pluggable storage backends, making it easy to use across different runtimes.

---

## Features

- Framework-agnostic core
- Express adapter
- Fastify adapter
- Memory and Redis stores
- Configurable request window
- Configurable request limit
- Custom key generation
- Standard `RateLimit` headers
- Legacy `X-RateLimit-*` headers
- `Retry-After` support
- Fail-open behavior
- Fully typed
- Production-ready architecture

## Architecture

```text
                 Request
                     │
          Express / Fastify Adapter
                     │
         evaluateRateLimit() Core Engine
                     │
      MemoryStore / RedisStore / Custom Store
```

The adapter extracts the client identifier, invokes the core engine, applies the generated headers, and either continues the request or returns a `429 Too Many Requests` response.

---

## Express

```ts
import { expressRateLimit, MemoryStore } from "@/blocks/rate-limit";

app.use(
  "/api",
  expressRateLimit({
    windowMs: 60_000,
    max: 100,
    store: new MemoryStore()
  })
);
```

## Fastify

```ts
import { fastifyRateLimit, MemoryStore } from "@/blocks/rate-limit";

);
```

## Configuration

| Option            | Type               | Default        | Description                    |
| ----------------- | ------------------ | -------------- | ------------------------------ |
| `windowMs`        | `number`           | `60000`        | Time window in milliseconds    |
| `max`             | `number`           | `100`          | Maximum requests allowed       |
| `message`         | `string \| object` | Error response | Response returned when blocked |
| `statusCode`      | `number`           | `429`          | HTTP status code               |
| `standardHeaders` | `boolean`          | `true`         | Send RFC RateLimit headers     |
| `legacyHeaders`   | `boolean`          | `true`         | Send X-RateLimit headers       |
| `keyGenerator`    | `(req) => string`  | Client IP      | Generate a unique identifier   |
| `store`           | `RateLimitStore`   | Required       | Storage implementation         |

## Response Headers

When enabled, the middleware automatically generates standard rate limit headers.

```http
RateLimit-Limit: 100
RateLimit-Remaining: 84
RateLimit-Reset: 42
```

Legacy compatibility headers are also supported.

```http
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 84
X-RateLimit-Reset: 1717344000
```

When a client exceeds the limit:

```http
Retry-After: 42
```

## Custom Key Generator

By default, the adapter identifies clients using their IP address. You can customize the key for authenticated APIs.

```ts
expressRateLimit({
  max: 100,
  windowMs: 60_000,
  store: new MemoryStore(),
  keyGenerator(req) {
    return req.user.id;
  }
});
```

## Custom Store

You can provide your own storage implementation by implementing the `RateLimitStore` interface.

```ts
import { RateLimitStore, RateLimitRecord } from "@/blocks/rate-limit";

class CustomStore implements RateLimitStore {
  async increment(key: string, windowMs: number): Promise<RateLimitRecord> {
    // Persist counter...
    return {
      hits: 1,
      resetTime: Date.now() + windowMs
    };
  }
}
```

This allows integration with databases, distributed caches, cloud KV services, or any custom storage backend.

## Why Blockend Rate Limiter?

There are many excellent rate limiting libraries available for Node.js, including `express-rate-limit`. Blockend isn't trying to replace them. Instead, it provides a framework-agnostic implementation that integrates naturally with the rest of the Blockend ecosystem while giving you complete ownership of the code.

**Choose Blockend when you want:**

- Framework-independent business logic
- Express and Fastify support from the same core
- Swappable storage backends
- Generated source code instead of hidden dependencies
- A consistent architecture across all Blockend blocks

---

## Framework-Agnostic Architecture

The rate limiting algorithm lives in a reusable core, while framework adapters are responsible only for integrating with Express or Fastify.

```text
Application
      │
      ▼
Express / Fastify Adapter
      │
      ▼
Rate Limiter Core
      │
      ▼
Memory Store / Redis Store / Custom Store
```

This separation makes the implementation easier to test, extend, and maintain.

## Flexible Storage

Start with an in-memory store during development.

```ts
import { MemoryStore } from "@/blocks/rate-limit";
const store = new MemoryStore();
```

Switch to Redis in production without changing your application logic.

```ts
import Redis from "ioredis";
import { RedisStore } from "@/blocks/rate-limit";

const redis = new Redis(process.env.REDIS_URL);
const store = new RedisStore(redis);
```

Need something else? Implement the `RateLimitStore` interface and use any storage backend you like.

## Which Should You Choose?

### Choose Blockend if you want:

- Express or Fastify support
- Framework-independent architecture
- Generated source code you fully control
- Easy storage migration
- Consistent APIs across Blockend blocks

### Choose `express-rate-limit` if you want:

- A ready-to-use Express-only middleware
- Minimal configuration
- An external dependency that you don't plan to customize

`express-rate-limit` is a mature and well-maintained library that works well for many applications. Blockend simply targets a different use case by focusing on ownership, extensibility, and portability.

## Production Recommendations

- Use Redis in production.
- Rate limit authentication endpoints more aggressively.
- Use authenticated user IDs instead of IP addresses when possible.
- Monitor rejected requests.
- Keep `standardHeaders` enabled for client visibility.
- Keep `legacyHeaders` enabled if supporting older clients.
- Combine with authentication, logging, and security middleware.
