The **Environment Configuration** block provides a centralized, type-safe way to validate your application's environment variables using Zod.

Instead of accessing `process.env` throughout your application, define your environment schema once, validate it during startup, and import the resulting `env` object anywhere in your project.

If any required variable is missing or invalid, the application immediately throws a formatted error before the server starts.

---

## Features

- Runtime validation powered by Zod
- Fully type-safe environment object
- Fail-fast startup validation
- Human-readable validation errors
- Supports default values
- Supports type coercion (numbers, booleans, etc.)
- Test-friendly by allowing custom environment sources
- Zero runtime overhead after initialization

## Creating Your Environment Schema

Start by loading your `.env` file.

```ts
import { config } from "dotenv";

config();
```

Next, define the variables your application expects.

```ts
import { z } from "zod";

const envSchema = z.object({
  PORT: z.coerce.number().default(5000),

  NODE_ENV: z.enum(["development", "production", "test"]).default("development"),

  DATABASE_URL: z.string(),

  JWT_SECRET: z.string().min(32)
});
```

Finally, validate and export your environment.

```ts
import { parseEnv } from "@/blocks/env-config";

export const env = parseEnv(envSchema);
```

If validation succeeds, `env` becomes a fully typed object inferred directly from your Zod schema.

---

## Using the Environment

Import the validated configuration anywhere in your application.

```ts
import { env } from "@/config/env";

console.log(env.PORT);

console.log(env.NODE_ENV);

await app.listen(env.PORT);
```

Unlike `process.env`, every property has the correct TypeScript type.

For example, after using `z.coerce.number()`:

```ts
env.PORT;
```

is typed as:

```ts
number;
```

instead of

```ts
string | undefined;
```

## Validation Errors

If validation fails, the application exits immediately with a readable error message.

Example:

```text
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ Environment Configuration Validation Failed
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

• DATABASE_URL → Required
• JWT_SECRET → String must contain at least 32 character(s)

Please verify your environment variables before restarting.
```

Failing during startup prevents unexpected runtime failures caused by missing configuration.

## Default Values

Use Zod defaults whenever a value is optional.

```ts
const schema = z.object({
  PORT: z.coerce.number().default(3000)
});
```

If `PORT` is missing, it automatically becomes:

```ts
3000;
```

---

## Type Coercion

Environment variables are always strings.

Use `z.coerce` to convert them into useful runtime types.

```ts
const schema = z.object({
  PORT: z.coerce.number(),

  ENABLE_CACHE: z.coerce.boolean()
});
```

Now both values are automatically parsed from strings.

For example:

```ts
PORT = 5000;
ENABLE_CACHE = true;
```

becomes

```ts
env.PORT;
// number

env.ENABLE_CACHE;
// boolean
```

---

## Testing

`parseEnv()` accepts an optional environment source, making it easy to test different configurations without modifying `process.env`.

```ts
const env = parseEnv(schema, {
  PORT: "3000",
  DATABASE_URL: "mongodb://localhost/test"
});
```

This is particularly useful for unit tests.

## A Note on Import Extensions

If your project uses `"module": "NodeNext"` (or `"Node16"`) in `tsconfig.json` together with `"type": "module"` in `package.json`, Node's ESM loader requires explicit file extensions on relative imports—`.js`, even though the source files are `.ts`. This is a Node.js requirement, not a Blockend one.

For example:

```ts
import { parseEnv } from "@/blocks/env-config/index.js";
```

or

```ts
import { env } from "./env.js";
```

instead of

```ts
import { parseEnv } from "@/blocks/env-config";
```

If you're unsure which applies to you, check `package.json` for `"type": "module"` and `tsconfig.json` for `"module": "NodeNext"` (or `"Node16"`). If neither is present, you don't need the `.js` extension.

## API

| Export               | Description                                                                                          |
| -------------------- | ---------------------------------------------------------------------------------------------------- |
| `parseEnv()`         | Validates an environment object against a Zod schema and returns a fully typed configuration object. |
| `EnvValidationError` | Error thrown when validation fails. Contains the original Zod validation issues.                     |

---

## Production Recommendations

- Load your `.env` file before calling `parseEnv()`.
- Validate your environment exactly once during application startup.
- Export a shared `env` object instead of accessing `process.env` throughout your application.
- Prefer `z.coerce.number()` and `z.coerce.boolean()` for numeric and boolean values.
- Use `.default()` for optional configuration with sensible defaults.
- Keep secrets (API keys, JWT secrets, database URLs) in environment variables rather than hardcoding them.
- Use descriptive validation messages for required values to make deployment failures easier to diagnose.
