The **Request Validator** block validates incoming HTTP requests before they reach your route handlers.

It provides a framework-agnostic validation core powered by Zod and lightweight adapters for supported frameworks. The core is responsible only for validation, while adapters integrate it with your web framework.

---

## Features

- Framework-agnostic validation core
- Powered by Zod with full type inference
- Validate request body, query parameters, and route parameters
- Automatically parses, coerces, and normalizes request data
- Access fully typed validated values via `validator.validated(req)`
- Express adapter included
- Optional integration with a global error handler
- TypeScript-first API

---

## Architecture

The Request Validator block is composed of two parts:

- **Core** — Validates arbitrary data using Zod. It has no dependency on Express or any other framework.
- **Adapter** — Integrates the core with your web framework by validating incoming requests before they reach your route handler.

When validation succeeds, the adapter:

- Parses the request using your Zod schemas.
- Applies defaults, coercions, and transforms.
- Replaces `req.body`, `req.query`, and `req.params` with the validated values.
- Exposes the validated payload through `validator.validated(req)` for fully typed access.

When validation fails, the adapter can either:

- Return a default `400 Bad Request` response.
- Forward the underlying `ZodError` to your application's global error handler.

## A Note on Import Extensions

If your project uses `"module": "NodeNext"` (or `"Node16"`) in `tsconfig.json` together with `"type": "module"` in `package.json`, Node's ESM loader requires explicit file extensions on relative imports — `.js`, even though the source file is `.ts`. This is a Node.js requirement, not a Blockend one.

If you're unsure which applies to you, check `package.json` for `"type": "module"` and `tsconfig.json` for `"moduleResolution": "NodeNext"`. If neither is present, you don't need the extension.

## Usage across frameworks

### Express

```ts
import { expressValidator } from "@blockend/request-validator/adapters/express";

app.post(
  "/users/:id",
  expressValidator({
    body: bodySchema,
    query: querySchema,
    params: paramsSchema
  }),
  (req, res) => {
    const { body, query, params } = expressValidator.validated(req);

    res.json({ body, query, params });
  }
);
```

### Hono

```ts
import { honoValidator, validated } from "@blockend/request-validator/adapters/hono";

app.post(
  "/users/:id",
  honoValidator({
    body: bodySchema,
    query: querySchema,
    params: paramsSchema
  }),
  (c) => {
    const { body, query, params } = validated(c);

    return c.json({ body, query, params });
  }
);
```

### Fastify

```ts
import { fastifyValidator } from "@blockend/request-validator/adapters/fastify";

const validator = fastifyValidator({
  body: bodySchema,
  query: querySchema,
  params: paramsSchema
});

fastify.post(
  "/users/:id",
  {
    preHandler: validator
  },
  async (request) => {
    const { body, query, params } = validator.validated(request);

    return { body, query, params };
  }
);
```

## Basic Usage

Define the parts of the request you want to validate.

```ts
import { z } from "zod";

const createUserValidator = expressValidator({
  body: z.object({
    name: z.string(),
    email: z.string().email(),
    phoneNumber: z.number()
  })
});
```

Register the validator before your route handler.

```ts
app.post("/users", createUserValidator, (req, res) => {
  const { body } = createUserValidator.validated(req);

  res.json(body);
});
```

The `validated(req)` helper returns the parsed request payload with full type inference. This includes any defaults, coercions, and transforms defined by your schemas.

Only the request sections you provide are validated. For example, if you only specify `body`, the validator ignores `query` and `params`.

---

## Validating Different Request Parts

You can validate any combination of the request body, query string, and route parameters.

```ts
const getUserValidator = expressValidator({
  body: z.object({
    name: z.string()
  }),

  query: z.object({
    page: z.coerce.number().default(1)
  }),

  params: z.object({
    id: z.uuid()
  })
});

app.get("/users/:id", getUserValidator, (req, res) => {
  const { body, query, params } = getUserValidator.validated(req);

  res.json({
    body,
    query,
    params
  });
});
```

## Validation Failures

By default, the Express adapter handles validation failures and returns a `400 Bad Request` response.

```json
{
  "success": false,
  "data": null,
  "message": "Validation failed",
  "errors": {
    "phoneNumber": ["Invalid input: expected number, received undefined"]
  }
}
```

This default behavior allows the validator block to work without requiring a global error handler.

---

## Using a Global Error Handler

If your application uses Blockend's Error Handler block, you can delegate validation errors to it instead.

```ts
const validator = expressValidator(createUserSchema, {
  propagateErrors: true
});

app.post("/users", validator, controller);
```

When `propagateErrors` is enabled, the adapter forwards the underlying `ZodError` to Express using `next(error)` instead of generating the response itself.

This is useful when you're using Blockend's Error Handler block to centralize API error responses.

---

## Using the Core Validator

The Express adapter is built on top of a framework-independent validator.

```ts
import { coreValidator } from "@/blocks/request-validator/core";

const validated = coreValidator(schema, data);
```

The core validator:

- Validates arbitrary input.
- Returns parsed data when validation succeeds.
- Throws a `ZodError` when validation fails.

Because it has no framework dependencies, it can be reused by adapters for other frameworks.

## API Reference

### `expressValidator(schema, options?)`

Creates Express middleware that validates the request body, query string, and route parameters using Zod.

The returned middleware also exposes a typed helper:

```ts
const validator = expressValidator(schema);

const { body, query, params } = validator.validated(req);
```

The returned values are fully parsed and typed according to your Zod schemas.

| Parameter                 | Description                                                                                                                |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| `schema`                  | Validation schema containing `body`, `query`, and/or `params`.                                                             |
| `options.propagateErrors` | When `true`, forwards the `ZodError` to Express instead of returning the default validation response. Defaults to `false`. |

---

## Best Practices

- Validate requests before they reach your controllers.
- Prefer `validator.validated(req)` instead of manually casting `req.body`, `req.query`, or `req.params`.
- Prefer `z.coerce` for query parameters and route parameters.
- Enable `propagateErrors` when using Blockend's Error Handler block.
- Keep validation schemas close to the routes that use them.
- Reuse validation schemas where appropriate.
